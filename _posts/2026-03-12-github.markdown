## GitHub概览

### 基本情况

- Git：
  - 语言上表示“自私的混蛋”、“脾气不好的人”，Linus Torvalds对自己的嘲讽。

  - 技术上表示采用**分布式版本控制系统**，分布式指开发者本地都有一个完整的仓库，服务器只是同步点。

- GitHub：
  - 由 GitHub, Inc. 开发，**2008** 年正式上线，2018 年被微软收购，现为微软旗下子公司（总部在加州旧金山）。

  - 截至 2025 年，GitHub 在全球拥有超过 **1.5 亿**开发者用户。平台上已托管了超过**4 亿**个代码仓库（包含公开和私有仓库），是目前世界上最大的源代码托管平台之一。


### 主要功能和特点

- **协作**机制：
  - **Issue**（问题跟踪）：可以在项目的 **Issue 区**提交 **Bug 报告、功能请求或讨论话题**。Issue 支持标签（label）、里程碑（milestone）和负责人分配等功能，方便对任务进行分类和跟踪。
  - **Pull Request**（拉取请求）：PR用于代码协作时提议合并代码更改。当开发者完成某个功能或修复bug后，可以创建 PR 请求将修改合并到主分支。团队成员可以在 **PR 界面**查看**代码差异**（diff），进行**代码评审**并讨论修改意见，确认无误后合并代码。Pull Request 是开源协作中常用的工作流程，有助于项目维护者审查外部贡献。
  - **Discussions**（讨论）：GitHub 提供讨论区功能，允许社区就项目相关话题进行交流，而不局限于具体的代码问题。这个功能为开放源代码项目的用户和维护者提供了一个交流的平台。
- **GitHub Pages**：GitHub 提供的**静态网站**（不运行后端程序、只提供已生成文件的网站）托管服务。用户可将仓库中的 **HTML、Markdown** 等内容通过 Pages 服务发布为网站。Pages 服务免费提供**\*.github.io**的子域名，支持自定义域名和 HTTPS，非常适合初学者用来展示项目成果。
- **GitHub Actions**：GitHub 内置的**自动化流程工具**。2019年正式推出，它允许用户在仓库内编写工作流（workflow）脚本，在仓库发生特定事件（如代码推送、创建 Pull Request 等）时自动触发指定的任务。通过 Actions，开发者可以实现**持续集成和持续部署（CI/CD）**流程，例如自动运行测试、构建应用、部署站点等，而无需借助第三方平台。GitHub 提供了托管的执行环境（如 **Linux、Windows、macOS 虚拟机**（能运行Xcode命令行工具））来运行这些任务，并提供了丰富的官方或社区制作的 **Actions 模板**，初学者可以方便地利用现成的方案来自动化项目流程。

### 主要用途

- **开源项目协作**：GitHub 最广为人知的用途就是作为开源社区协作的平台。全球大量的开源软件项目托管在 GitHub 上，开发者通过 Fork 项目、提交 Pull Request 等方式为开源项目做贡献。GitHub 提供的 issue 和 PR 等机制使得分布在世界各地的开发者可以一起讨论问题、改进代码。这使它成为开源协作的首选平台之一。（例如网络框架 **Alamofire**、图像加载库 **SDWebImage** 等热门库）
- **私有项目托管**：GitHub 也被广泛用于私有代码的托管。个人开发者或企业团队可以创建私有仓库，将自己的专有代码、安全地储存在 GitHub 上。仓库的可见性可以设置为私有，只有被邀请的协作者才能访问代码。这对于团队开发内部项目或个人保存自己的代码项目非常有用。GitHub Free 账户如今也支持创建不限数量的私有仓库，这降低了入门门槛。
- **团队项目协同**：GitHub 为企业和团队提供了强大的协同开发支持，包括组织账户（**Organization**）和**团队权限管理**等功能。团队成员可以在同一个仓库下协作开发，利用 Issue 来分配任务，利用 Pull Request 来审查和合并代码。由于协作功能完善，许多公司将 GitHub作为代码开发流程的一部分。据统计，超过 90% 的全球财富100强企业都在使用 GitHub 来托管和协作他们的代码。可见，无论是小型创业团队还是大型企业，GitHub 都已成为主流的协同开发平台。
- **项目展示与作品集**：通过 GitHub，开发者可以方便地展示自己的项目成果。一方面，公开仓库本身就是个人作品的展示窗口——其他人可以浏览你的代码、提交记录和项目Star数等，从而了解你的开发能力。另一方面，利用 GitHub Pages 服务，可以为项目创建精美的说明文档或产品网站，将项目的用途、截图、教程等展示出来。这对于开源项目吸引用户和贡献者、或个人作为求职作品集，都是非常有帮助的。

### 使用方法

- **上传代码**：

  - 使用 **Git 命令行**推送：这是开发者常用的方法。首先在本地电脑上进入你的项目文件夹，通过 _**git init**_ 将其初始化为 Git 仓库，然后使用 _**git remote add origin 仓库地址**_ 将本地仓库与 GitHub 上的新仓库关联。接下来，将代码变更提交到本地仓库（_**git add .**_ 添加文件，_**git commit**_ 提交记录），最后使用 _**git push -u origin main**_ 将本地提交推送到 GitHub 的远程仓库。

- **使用Pull Request** 和 **Issue 追踪**：

  - **发起 PR**：PR 是向他人的仓库提议代码更改的流程，也可用于在自己仓库中让团队成员审阅代码。假设您在某项目的某个分支做了一系列修改，现在希望将它们合并到主分支（main）。这时您可以**打开该仓库，切换到您的分支，点击 New Pull Request** 按钮来创建拉取请求。在 PR 页面填写标题和描述，说明您的更改内容。创建后，项目的维护者会收到通知，他们可以查看您提交的代码差异并在页面上与您讨论。一旦维护者审核通过，他们会合并（Merge）您的 Pull Request，这样您的修改就正式进入了项目代码库。如果有需要修改的地方，也可以在 PR 下评论要求更正，您更新代码后 Pull Request 会自动显示最新的提交。
  - **提出和处理 Issue**：Issue 用于跟踪任务、Bug 或改进建议。对于**自己的仓库**，您可以在 **Issues 标签页点击 New issue** 来新建一个 Issue，填写标题和描述来描述问题或建议。您也可以在参与**他人的项目**时**提交 Issue**，报告发现的错误或提出功能请求。Issue 列表就像项目的任务列表，维护者会根据情况回应每个 Issue。例如，他们可能会针对 **Bug** 的 Issue **讨论解决方案、标记优先级，完成后在评论中注明修复**并关闭 Issue。作为项目维护者，您应及时查看和处理 Issue：可以在讨论区回复提问、标签分类（如标注为“bug”或“enhancement”）、指派负责的人，并在问题解决后关闭 Issue。GitHub 的 Issue 功能支持**富文本和图片**，方便描述问题细节，也支持引用 commit 或 Pull Request 来关联代码改动记录。

### 开源代码许可证（GPT5 - DR）

- 开源许可证是在开源项目中用于规定他人如何使用、修改和分发代码的法律协议。

- 开源许可证大致分为两大类：宽松许可证（**Permissive**）和**Copyleft**许可证（也称传染性许可证）。

- **宽松许可证**对代码的使用和再发布限制最少，允许他人在几乎任何环境下自由使用、修改和闭源分发衍生作品，并通常只要求保留**版权声明和许可证文本**。典型的宽松许可证有 **MIT、Apache 2.0、BSD** 等。

- **Copyleft许可证**则要求在一定范围内保持源代码开放：一旦对包含该许可证的代码进行修改并发布，就必须以相同的许可证开源发布…**强Copyleft**（如 **GPL**）要求**整个衍生作品**都采用相同许可证公开源代码，而**弱Copyleft**（如 **LGPL、MPL**）则只要求对**特定部分**（如库或修改过的文件）开源。

- MIT 许可证：

  - MIT 许可证是最宽松和简单的开源许可证之一，被广泛用于各类开源项目。
  - 你需要在分发的软件中附上原始的**版权声明和许可文本**。
  - MIT 许可证包含**免责条款**，不提供任何担保，作者不对代码使用造成的问题负责。
  - 据 GitHub 2025 年的数据，MIT 许可证约占**1/3**。

- Apache 2.0 许可证：

  - Apache 软件基金会发布的一款著名开源许可证。
  - 允许将代码用于商业用途和闭源项目，也允许自由修改和再发布，而且发布时可以选择任意许可证。
  - 分发含有 Apache 2.0 代码的软件时，必须**附带许可证文本和版权声明**，并在**修改过的文件中标明你做过修改**。
  - 你不得使用原作者或 Apache 基金会的商标名称来宣传你的派生产品，以免暗示得到官方背书。
  - Apache 2.0 明确包含**专利授权条款**。授权者通过该许可证授予使用者必要的专利权利，且规定如果使用者就所获得的软件专利提起诉讼，授权将终止。这一点对企业来说非常重要：它降低了因专利问题使用开源代码的法律风险。相比之下，MIT 等许可证没有明示专利授权，这也是许多大型公司更偏好 Apache 2.0 的原因之一。
  - Apache 2.0 许可证非常适合**商业和企业级项目**。
  - 近年来 Apache 2.0 在开源社区的使用率一度超过 MIT，成为最受欢迎的许可证之一。

- GPL 许可证 (GNU 通用公共许可证)：

  - GNU GPL是最具代表性的强 Copyleft（传染性）许可证，由**自由软件基金会**的理查德·斯托曼创建，用以确保软件及其衍生版本始终以**开源**方式发布。
  - GPL 的宗旨是在允许他人自由使用、修改代码的同时，强制要求凡是基于 GPL 代码开发并发布的软件也必须采用 GPL 协议开源发布。
  - GPL 许可证最主要的触发条件是**发布/分发**。如果你仅在内部或个人使用 GPL 代码（不向公众发布二进制或源码），那么法律上并不强制你公开源代码。
  - GPL 许可更适合那些强调开源精神、希望保护开源生态的项目。

- LGPL 许可证 (GNU 宽松通用公共许可证)：

  - LGPL（Lesser GPL 或 Library GPL）被称为“宽松版”的 GPL。它最初设计用于库（Library），目的是允许这些开源库被闭源的软件所使用，而不强制整个闭源软件开源。
  - 如果你修改了 LGPL 的开源库本身，你仍须将**修改后的库部分开源**。
  - LGPL 常用于各种希望被广泛采纳的开源库或框架。

- BSD 许可证：

  - BSD 许可证是一系列源自Berkeley Software Distribution的宽松开源许可证。
  - 常见的有 BSD 3-Clause（新BSD许可证）和 BSD 2-Clause（简化BSD许可证）。
  - **BSD 3-Clause** 许可证包含三条主要条件：
    - 保留**版权声明和许可声明**：无论以源码还是二进制形式分发，都必须保留原作者的版权和许可免责声明。
    - **免责条款**：和 MIT 类似，提供“按原样”无担保的声明，作者不对软件承担责任。
    - **不使用作者名义推广**：除非获得书面许可，不能使用原开源项目的贡献者名称、机构名称或项目名称为衍生产品做市场推广。
  - BSD 2-Clause 则在3-Clause基础上去掉了第三条限制。
  - 它非常适合不希望限制他人使用，又希望保留署名权和免责的项目。
  - BSD 许可证也是 GitHub 上常见的许可证类型之一，其使用率虽然不及 MIT 和 Apache，但一直保持稳定流行。
